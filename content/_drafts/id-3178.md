JavaScript, at least at it's core, is synchronous in nature. It's thread is singular, which means it can only do one thing at a time. When a function is invoked in our application it is added to the Call Stack, executed (last in, first out - like a stack of plates), and removed from the Call Stack on completion/return of that function. With that, in our program, it will not continue down and execute code in the Global Execution Context while simultaneously executing code in the Local Execution Context of the `animateHeight` (and `slideSomeMore`) function.

You can see a visual demonstration of this by removing the `setTimeOut` in _framework.js_ on [line 121](https://github.com/automattic-hiring/frontend-code-test-dr-5743/blob/master/framework.js#L121) and just making a recursive call to `slideSomeMore` directly. In doing so, you'll notice that the browser is completely halted until the condition in the `while` loop on [line 115](https://github.com/automattic-hiring/frontend-code-test-dr-5743/blob/master/framework.js#L115) is met and the program is able to continue on with the execution of the code below. At that point the "animation" has already completed and just jumps into the end position.

This is where `setTimeOut` comes into play. The `setTimeOut` function is not a part of the core JavaScript language, rather a method from the Web API (under [WindowOrWorkerGlobalScope](https://developer.mozilla.org/en-US/docs/Web/API/WindowOrWorkerGlobalScope)) that JavaScript interfaces with. It is in interfacing with these external Web APIs that brings asynchronicity to JavaScript.

In our `slideSomeMore` function, when we reach line 121, the `setTimeOut` function is executed. That function doesn't do anything particularly meaningful in JavaScript itself, however, so our program continues on with it's normal execution. In the Web API, however, it starts off a timer. When the timer is complete (in this case at 0ms) it fires off the callback function (another call to `slideSomeMore`, in our case here).

So how (and when) does the callback function from the timer here make it's way back into the Call Stack?

Once a callback is ready to be executed, after the completion of the Web API process, it's added to a Callback Queue. From here, it waits to eventually be added back onto the Call Stack via the Event Loop. The Event Loop is a process that runs a continuous check to see if the Call Stack is empty. Only when the Call Stack is empty, meaning that our program has completed the execution of all of the synchronous code, will the Event Loop move the callback function from the Callback Queue on to the Call Stack to be executed.

With this process, the timer duration we pass as an argument to `setTimeOut` is not something we can rely on as a time of execution. It's would be better thought of as a minimum time until execution. This is because, if we have a process being executed on the Call Stack that takes a longer time to complete, our callback from `setTimeOut` will be delayed in the Callback Queue until it completes, the Call Stack is empty, and it can be moved over via the Event Loop.

In addition, if our callback does not make it in time for the next repaint, it will be delayed until the next cycle of the Event Loop. With regard to animations, this is far from ideal. If we lose one frame, then the animation will be performed twice in the next repaint. This causes a visibly clunky animation, like you see in our case with the Expand/Collapse toggle.

Lucky for us, the Web API has a newer method available to us called `requestAnimationFrame`. With this method, the browser can optimize our animation steps together into a single reflow and repaint cycle, resulting in a much smoother animation - which you can see in this PR. While newer than `setTimeOut`, `requestAnimationFrame` has still been around for some time and [browser support is very good](https://caniuse.com/#feat=requestanimationframe).
